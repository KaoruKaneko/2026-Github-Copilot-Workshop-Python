# Pomodoro Timer 段階的実装計画

このドキュメントは、UIモック、アーキテクチャ、機能一覧を踏まえた段階的な実装計画を定義する。

参照資料:
- architecture.md
- features.md
- pomodoro.png

## 1. 実装粒度の方針
- 1フェーズは「2〜3日で完了する縦切り」を基本とする
- 各フェーズで UI + ロジック + 最小テスト まで実装する
- 各フェーズに明確な完了条件（DoD）を設定する
- ロジックのみの先行実装を避け、毎回動作可能な形で統合する

## 2. フェーズ計画

### フェーズ0: 土台作成（0.5〜1日）
目的:
- 開発の土台を整え、最小の実行可能状態を作る

実装内容:
- Flaskアプリの起動
- 画面配信ルートの作成
- templates/staticの初期構成
- テスト実行基盤（バックエンド/フロント）導入

完了条件:
- ブラウザでトップ画面が表示される
- テストコマンドが実行できる

### フェーズ1: タイマーMVP最小動作（1〜2日）
目的:
- Focusタイマーの基本操作を成立させる

実装内容:
- 開始/一時停止/再開/リセット
- 残り時間（mm:ss）表示
- 終了予定時刻ベースの残り時間計算
- フロント状態管理（running/paused等）

テスト:
- タイマー計算ユニットテスト
- 基本状態遷移ユニットテスト

完了条件:
- Focusタイマーの一連操作が安定動作する

### フェーズ2: 3モード化と遷移ルール（1〜2日）
目的:
- Focus/Short Break/Long Break の運用を成立させる

実装内容:
- モード切替UI
- Focus完了回数カウント
- Long Break intervalに基づく自動遷移
- スキップ操作の反映

テスト:
- モード遷移テスト
- Long Break切替境界テスト

完了条件:
- 1サイクル通しの遷移が正しく機能する

### フェーズ3: 設定管理と永続化（1〜2日）
目的:
- 設定変更と復元を可能にする

実装内容:
- 設定UI
- GET /api/settings
- PUT /api/settings
- 設定バリデーション
- 設定値のJSONファイル保存（代替: localStorage保存）

テスト:
- 設定APIの正常系/異常系
- 設定境界値テスト

完了条件:
- 設定保存後に再読み込みしても値が復元される

### フェーズ4: セッション履歴保存（1日）
目的:
- 完了/中断セッションを記録し、統計の基礎データを作る

実装内容:
- POST /api/sessions
- SessionLog保存
- 中断理由の記録

テスト:
- 保存項目整合性テスト
- API異常系テスト

完了条件:
- セッション実績が必要項目を満たして保存される

### フェーズ5: 統計表示（1日）
目的:
- 履歴を可視化し、利用価値を高める

実装内容:
- GET /api/stats/daily
- GET /api/stats/weekly
- 当日完了数/集中時間の表示

テスト:
- 集計ロジックユニットテスト
- APIレスポンステスト

完了条件:
- 履歴と統計表示の値が一致する

### フェーズ6: 品質強化（1〜2日）
目的:
- 実利用時の安定性と使いやすさを向上させる

実装内容:
- API失敗時のエラーハンドリング
- 通知音ON/OFF
- レスポンシブ対応
- アクセシビリティ改善

テスト:
- 非アクティブタブ復帰時の挙動
- pause/resume連続操作
- エラー時のフォールバック

完了条件:
- 主要異常系でも操作不能にならない

### フェーズ7: テスト拡張とリリース準備（1〜2日）
目的:
- 回帰防止とリリース耐性を確保する

実装内容:
- Clock抽象注入テスト
- Repository差し替えテスト（File/InMemory）
- API契約テスト
- 最小E2E（1サイクル）

完了条件:
- 主要機能の自動回帰検知が可能

## 3. 共通DoD（各フェーズ共通の完了定義）
- ユーザー操作シナリオが最低1本成立している
- 追加ロジックに対するユニットテストが存在する
- API追加時は正常系と異常系を確認している
- 既存主要機能に回帰がないことを確認している
- 次フェーズへ持ち越す課題を明文化している

## 4. 粒度の具体例
適切な粒度:
- タイマー開始機能（UIボタン、状態更新、時間表示、ユニットテスト）

粗すぎる粒度:
- タイマー機能全体を一括実装

細かすぎる粒度:
- ボタン見た目のみ、ロジックのみ、表示のみを長期間分断

## 5. 推奨マイルストーン
- マイルストーン1: フェーズ0〜2
  - ゴール: タイマーとして成立する動作を完成
- マイルストーン2: フェーズ3〜5
  - ゴール: 保存と統計を備えた実用レベルに到達
- マイルストーン3: フェーズ6〜7
  - ゴール: 品質を高めてリリース可能状態にする

## 6. 備考
- 本計画は小さく動く成果を積み上げることを重視する
- 仕様変更が発生した場合は、次フェーズ開始前にDoDを再定義する
